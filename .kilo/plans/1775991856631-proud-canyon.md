# Plan: Preview Outputs After Shuffling / Beat Join

## Problem

The studio UI computes shuffle queues and beat-join arrangements but does **not** generate or display a playable preview video after these operations. The current `runProcess()` flow calls `/api/preview/section` with a single time window from one segment — it never renders the rearranged clip sequence. The right panel's "Prepared Preview" section either shows the placeholder text or a single raw segment slice, not the actual shuffled/joined output.

The user wants to **see and hear the result** after performing shuffle or beat-join operations — a playable video preview that reflects the reordered segment sequence.

## Critical Constraint: Vercel Deployment

The project deploys to Vercel. This means:
- **No FFmpeg binary** is available in serverless functions
- **No persistent filesystem** — temp files are lost between invocations
- **Execution timeout** — 10s default, 60s on Pro
- The current `previewGeneration.ts` and `mediaProbe.ts` both depend on local FFmpeg — they already **do not work on Vercel**

This plan introduces a **dual-mode preview architecture**:
1. **Browser-side preview** (works everywhere, including Vercel): Uses the HTML5 `<video>` element + `MediaSource` API or simple sequential clip playback to construct a preview from browser-accessible video URLs
2. **Server-side FFmpeg preview** (local/dev only): The existing FFmpeg pipeline, enhanced with concat support, available when running locally with `.local-fixtures` media

## Current State

### What works
- **Segment manifest** (`segmentManifest.ts`): builds music-cut-driven segments with motion descriptors
- **Shuffle queue** (`shuffleQueue.ts`): produces a reordered clip index array (`clipOrder`)
- **Manifest ranking** (`manifestRanking.ts`): ranks candidates by motion/color/random mode
- **Beat join** (`BeatJoinTab.tsx`): builds arrangement segments from audio analysis with stutter presets
- **Section recompute state machine** (`sectionRecompute.ts`): idle → stale → recomputing → ready → swapped
- **Preview generation** (`previewGeneration.ts`): FFmpeg-backed single-segment clip extraction (local only)
- **Preview asset serving** (`/api/preview/asset`): serves generated MP4 files (local only)
- **Right panel** (`StudioRightPanel.tsx`): has a `<video>` element that plays when `previewAssetUrl` is set

### What's broken / missing
1. `runProcess()` sends only `startTime`/`endTime` for a single segment — it never sends the reordered clip list
2. No mechanism to play back a reordered sequence of video clips in the browser
3. The shuffle and beat-join tabs compute `clipOrder` / `arrangementSegments` purely for display; the preview pipeline ignores them
4. Beat join's `arrangementSegments` are local to `BeatJoinTab.tsx` — they never flow back to `StudioApp` for preview
5. The FFmpeg preview path is local-only and will not work on Vercel

## Implementation Plan

### Step 1: Browser-side sequential clip preview

**New file**: `src/components/studio/previewPlayer.ts`

Create a browser-side preview orchestrator that:
1. Accepts an ordered list of `{ videoUrl: string; startTime: number; endTime: number }` segments
2. Creates a hidden `<video>` element
3. Plays segments sequentially: loads clip → seeks to start → plays → on `timeupdate` when past `endTime`, loads next clip → repeat
4. Provides callbacks for progress (current segment index, playback time) and completion
5. Handles edge cases: zero-duration segments, missing URLs, last segment

This is the **primary preview mechanism** that works on Vercel because it uses only browser APIs and the video object URLs already available from upload.

### Step 2: Create a visible preview player component

**New file**: `src/components/studio/PreviewPlayer.tsx`

A React component that:
1. Renders a `<video>` element visible in the right panel or inline
2. Shows play/pause controls, current segment indicator, progress bar
3. Accepts the ordered segment list and orchestrates playback via `previewPlayer.ts`
4. Shows the recompute/progress state while building
5. Displays segment labels (e.g., "Playing SEG_03 → SEG_01 → SEG_05") during preview

### Step 3: Lift arrangement segments out of BeatJoinTab

**New file**: `src/components/studio/arrangementBuilder.ts`

Move `buildArrangementSegments()` and related helpers from `BeatJoinTab.tsx` into a shared module. This allows `StudioApp` to compute the arrangement segments and use them for preview.

**Modify**: `src/components/studio/panels/BeatJoinTab.tsx` — remove local `buildArrangementSegments`, accept `arrangementSegments` as a prop

**Modify**: `src/components/StudioApp.tsx` — compute arrangement segments in useMemo, pass down to BeatJoinTab

### Step 4: Build the preview segment list from each tab

**Modify**: `src/components/StudioApp.tsx`

Add a new `useMemo` that computes the preview segment list based on `tab`:

```ts
const previewSegments = useMemo(() => {
  switch (tab) {
    case "shuffle":
    case "join": {
      // Map effectiveClipOrder + workingBeatSplitSegments + videoSources
      // to [{ videoUrl, startTime, endTime }, ...]
      return effectiveClipOrder.map(clipId => {
        const segment = workingBeatSplitSegments[clipId];
        const source = videoSources[segment.sourceClipIds[0]];
        return {
          videoUrl: source?.videoUrl,
          startTime: segment.start - (source timeline offset),
          endTime: segment.end - (source timeline offset),
          label: `SEG_${clipId + 1}`
        };
      }).filter(s => s.videoUrl);
    }
    case "beatjoin": {
      // Map arrangementSegments to preview segments
      return arrangementSegments.map(seg => {
        const source = videoSources[seg.clipId];
        return {
          videoUrl: source?.videoUrl,
          startTime: seg.start,
          endTime: seg.end,
          label: seg.detailLabel
        };
      }).filter(s => s.videoUrl);
    }
    default:
      return [];
  }
}, [tab, effectiveClipOrder, workingBeatSplitSegments, videoSources, arrangementSegments]);
```

The key challenge is mapping from the virtual timeline (where all source clips are concatenated) back to per-source timestamps. Each segment's `start`/`end` is in the virtual timeline; we need to subtract the source clip's virtual offset to get the local timestamp within that source video.

### Step 7: Add source clip time offset tracking

**Modify**: `src/components/studio/sourceTimeline.ts`

Export a helper function `getSourceClipTimeOffset(sourceClips, clipId)` that returns the virtual timeline offset for a given source clip. This is needed to convert virtual-timeline segment times back to per-source local times.

### Step 8: Add FFmpeg concat preview for local dev

**Modify**: `src/components/studio/previewGeneration.ts`

Add `generateConcatPreview()` that:
1. Accepts an ordered list of `{ inputPath, startTime, endTime }` segments
2. Re-encodes each segment to a common format (same settings as existing single-segment preview)
3. Writes an FFmpeg concat demuxer file
4. Runs FFmpeg concat to produce a single merged MP4
5. Returns a `GeneratedPreviewAsset`

**Modify**: `src/app/api/preview/section/route.ts`

Add support for a `segments[]` payload that routes to `generateConcatPreview()`.

This path is **local/dev only** — it requires FFmpeg and local file access. On Vercel, the browser preview path is used instead.

### Step 9: Auto-detect preview mode

**Modify**: `src/components/StudioApp.tsx`

Detect whether FFmpeg is available (try a lightweight probe on mount, or check for local fixture availability). Set a flag `canUseFfmpegPreview`:
- If FFmpeg is available and local paths exist: use FFmpeg concat preview
- Otherwise: use browser sequential preview

This ensures the same codebase works both locally (with FFmpeg previews) and on Vercel (with browser previews).

## Architecture Decision: Browser Preview vs FFmpeg Preview

| Concern | Browser Sequential Preview | FFmpeg Concat Preview |
| --- | --- | --- |
| Works on Vercel | Yes | No |
| Quality | Source quality | Re-encoded (configurable) |
| Latency | Instant (no processing) | Seconds to minutes |
| Gapless transitions | Small gap between clips | Seamless |
| Audio sync | Per-clip, may drift at joins | Guaranteed by FFmpeg |
| Complexity | Low | Medium |

**Strategy**: Use browser sequential preview as the default (works everywhere). Offer FFmpeg concat preview as an upgrade path for local dev / future desktop version where seamless transitions matter.

## Risks

| Risk | Mitigation |
| --- | --- |
| Browser sequential preview has small gaps between clips | Acceptable for preview; FFmpeg path available locally |
| Virtual-to-local time mapping may be wrong for multi-source clips | Test with real uploads; use `getSourceClipTimeOffset()` |
| Beat join arrangement segments reference clips differently than segment manifest | Map via `clipId` index consistently |
| FFmpeg not available on Vercel | Browser preview is the primary Vercel path |
